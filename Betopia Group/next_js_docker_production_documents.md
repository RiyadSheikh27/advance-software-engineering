# Betopia Limited — Dockerized Next.js Deployment

**Domain:** betopialimited.com
**Server:** ip-172-31-30-238 (public IP: 47.130.141.143)
**Folder:** `~/betopia-limited-client`
**Container name:** `betopia-limited`
**Port:** 6002 (host and container)
**Runtime:** Docker CLI emulated via Podman (rootless) — `docker` commands below all actually run through Podman on this server.

This app is deployed differently from the rest of the Betopia stack (which uses PM2 directly). This one ships with its own `Dockerfile` and runs as a container, so PM2 is **not** used to manage its process — only Docker/Podman is.

---

## Why Docker instead of PM2 here

`betopia-limited-client` came with a pre-built multi-stage `Dockerfile` (deps → builder → runner) using the Next.js `output: "standalone"` build. Since the project was already containerized, deployment followed that pattern instead of converting it to a plain PM2 `next start` process like the other apps.

---

## Key Files

| File | Purpose |
|---|---|
| `Dockerfile` | Multi-stage build: installs deps, builds the app, copies the standalone output into a minimal `node:22-alpine` runner image. |
| `next.config.mjs` | Must contain `output: "standalone"` — required for this Dockerfile's `COPY --from=builder /app/.next/standalone ./` step to work. Also contains `serverActions.allowedOrigins`, which must list the production domain (see Gotchas). |
| `.env` | Runtime environment variables, passed into the container via `--env-file` at `docker run`. Does **not** get baked into the image — must be supplied every time the container is (re)created. |
| `.dockerignore` | Keeps `node_modules`, `.git`, `.next`, and unrelated utility scripts (`*.py`, `extract_data.js`, etc.) out of the build context. |
| `package.json` | Standard Next.js scripts. Not directly used at container runtime — the container runs `node server.js` (the standalone server) directly, bypassing `npm start` entirely. |

## Dockerfile — annotated

```dockerfile
# Dependency stage — installs node_modules
FROM node:22-alpine AS deps
RUN apk add --no-cache libc6-compat
WORKDIR /app
COPY package.json ./
RUN npm config set legacy-peer-deps true && \
    npm install && \
    npm update caniuse-lite browserslist && \
    npm install --package-lock-only

# Builder stage — runs the Next.js build
FROM node:22-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY --from=deps /app/package-lock.json ./
COPY . .
ENV NEXT_TELEMETRY_DISABLED=1
RUN npm run build

# Runner stage — minimal final image, only the standalone output
FROM node:22-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1
RUN addgroup --system --gid 1001 nodejs
RUN adduser --system --uid 1001 nextjs
COPY --from=builder /app/public ./public
RUN mkdir .next && chown nextjs:nodejs .next
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
USER nextjs
EXPOSE 6002
ENV PORT=6002
CMD ["node", "server.js"]
```

> Originally shipped with `EXPOSE 6001` / `ENV PORT=6001`, which collided with `betopia-group-client` (also on 6001). Changed to 6002 for this deployment.

## ⚠️ Critical Gotcha — `HOSTNAME` env var breaks the bind address

**This is the bug that caused the deployment to silently fail with `Connection refused` despite the container reporting `✓ Ready` in its logs.**

The standalone `server.js` generated by Next.js binds using:
```js
const hostname = process.env.HOSTNAME || '0.0.0.0'
```

Docker/Podman containers automatically set `HOSTNAME` to the container ID (e.g. `8e40415b637b`) unless overridden. That auto-set value gets picked up by `server.js` **instead of** the `'0.0.0.0'` fallback, causing the server to bind only to that specific internal address rather than all interfaces — making it unreachable from the host, from `localhost` inside the container, and from Nginx.

**Symptom:** `docker logs` shows a clean `✓ Ready` with no errors, `docker ps` shows the container `Up`, but `curl` (from host or even from inside the container via `docker exec`) returns `Connection refused` / `Recv failure: Connection reset by peer`.

**Diagnosis:**
```bash
docker exec -it betopia-limited sh -c 'echo $HOSTNAME'
# if this prints a container ID hash instead of 0.0.0.0, this is the bug

docker exec -it betopia-limited sh -c "cat /proc/net/tcp"
# local_address column will show the bound IP in hex instead of 00000000 (wildcard)
```

**Fix — always pass this explicitly on every `docker run`:**
```bash
-e HOSTNAME=0.0.0.0
```

This is now a permanent part of the run command below. Don't omit it on any future recreate.

## Other gotchas hit during this deployment

1. **`.env` must exist before `docker run --env-file`** — the flag fails hard (not silently) if the file path doesn't exist. Confirm first: `ls -la ~/betopia-limited-client/ | grep -i env`.
2. **Container name conflicts** — re-running `docker run` with the same `--name` after a previous attempt (even a failed/exited one) fails with "name already in use." Always `docker rm -f betopia-limited` before recreating.
3. **`localhost` vs `127.0.0.1` vs `0.0.0.0` in curl tests** — Podman's rootless port forwarder (`pasta`) only reliably forwards IPv4. `curl http://localhost:6002` can fail with `Recv failure: Connection reset by peer` (IPv6 `::1` attempted first) even when the app is fully healthy. Always test with `curl http://127.0.0.1:6002` on this host, and point Nginx's `proxy_pass` at `127.0.0.1`, not `localhost`.
4. **`serverActions.allowedOrigins` in `next.config.mjs`** — must include the production domain or Server Actions (form submissions / mutations) from `betopialimited.com` will be rejected with a forbidden-origin error once live. Confirmed present as:
   ```js
   serverActions: {
     allowedOrigins: ["*.ngrok-free.app", "*.loca.lt", "localhost:6002", "betopialimited.com"],
   },
   ```
   Add `www.betopialimited.com` too if the `www` subdomain is used.

## Build & Run — full sequence

```bash
cd ~/betopia-limited-client

# Build the image
docker build -t betopia-limited .

# Remove any existing container with this name (safe even if none exists... almost; ignore "no such container" errors)
docker rm -f betopia-limited

# Run it
docker run -d \
  --name betopia-limited \
  --restart unless-stopped \
  -p 6002:6002 \
  -e HOSTNAME=0.0.0.0 \
  --env-file ~/betopia-limited-client/.env \
  betopia-limited
```

## Restarting / Redeploying After Code Changes

Unlike the PM2 apps, `pm2 restart` does not apply here. A code change requires a **full image rebuild**, since the build output is baked into the image at build time, not read live from disk.

```bash
cd ~/betopia-limited-client
git pull

# Rebuild the image
docker build -t betopia-limited .

# Stop and remove the running container
docker rm -f betopia-limited

# Recreate it from the new image
docker run -d \
  --name betopia-limited \
  --restart unless-stopped \
  -p 6002:6002 \
  -e HOSTNAME=0.0.0.0 \
  --env-file ~/betopia-limited-client/.env \
  betopia-limited
```

> If only `.env` changed (no code change), you still need to `docker rm -f` and re-run with the updated `--env-file` — env vars are injected at container creation time, not read live. A rebuild of the image is **not** needed in that case, only a recreate of the container.

**Quick reference table:**

| What changed | Rebuild image? | Recreate container? |
|---|---|---|
| App code / dependencies | Yes (`docker build`) | Yes |
| `.env` values only | No | Yes |
| `Dockerfile` itself | Yes | Yes |
| Nginx config | No | No (just reload nginx) |

---

## After VPS Reboot

`--restart unless-stopped` means Podman should bring the container back up automatically on boot, the same way PM2's `pm2 resurrect` does for the other apps. Verify after any reboot:

```bash
docker ps -a | grep betopia-limited
```

If it's not running:
```bash
docker start betopia-limited
```

If it's missing entirely (rare, but possible if Podman's state got reset), recreate it using the full `docker run` command above.

## Nginx Config

**File:** `/etc/nginx/sites-available/betopia-limited`, symlinked into `/etc/nginx/sites-enabled/`

```nginx
server {
    listen 80;
    server_name betopialimited.com www.betopialimited.com;

    location / {
        proxy_pass http://127.0.0.1:6002;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_cache_bypass $http_upgrade;
    }
}
```

> Uses `127.0.0.1` explicitly, not `localhost` — see gotcha #3 above. Certbot will auto-append an SSL `server` block and HTTPS redirect to this same file after running the command below.

```bash
sudo nano /etc/nginx/sites-available/betopia-limited
sudo ln -s /etc/nginx/sites-available/betopia-limited /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx
```

> ⚠️ Same rule as the rest of the Betopia stack: symlinking into `sites-enabled/` does nothing until you reload. Always `nginx -t && systemctl reload nginx` right after.

## Verification Checklist (in order — don't skip ahead)

Run these in sequence any time you redeploy. Each step isolates a different layer, so if something's broken you'll know exactly where.

```bash
# 1. Container running, not crash-looping
docker ps
# expect: STATUS = "Up X seconds/minutes", not "Restarting"

# 2. Clean startup logs
docker logs betopia-limited --tail 50
# expect: "✓ Ready" with no error trace after it

# 3. Reachable on the container's own port, from the host
curl -I http://127.0.0.1:6002
# expect: HTTP/1.1 200 OK, X-Powered-By: Next.js

# 4. Reachable via Nginx, by IP with explicit Host header (works even before DNS propagates)
curl -I -H "Host: betopialimited.com" http://47.130.141.143

# 5. Reachable via the real domain, HTTP
curl -I http://betopialimited.com

# 6. Reachable via the real domain, HTTPS (after certbot)
curl -I https://betopialimited.com
```

Only proceed to the next step once the current one returns a clean `200 OK` with the Next.js headers. This order matches exactly how this deployment was actually debugged — the bug (gotcha #1 above) was only caught because step 3 was checked before moving to Nginx.
